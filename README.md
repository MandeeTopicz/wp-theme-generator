# WP Theme Generator

An AI-powered WordPress Block Theme generator that turns natural language descriptions into complete, installable WordPress themes with live preview.

Built as an Automattic coding challenge submission, this project explores how generative AI can lower the barrier to WordPress Full Site Editing (FSE) theme development. Instead of requiring knowledge of theme.json schemas, block markup syntax, and template hierarchy, users describe what they want in plain English and receive a production-ready block theme — complete with style variations, validated markup, and a live WordPress preview powered by WordPress Playground. The goal aligns with WordPress's core mission: democratizing publishing by making theme creation accessible to anyone.

## What It Does

- **Natural language to complete WordPress Block Theme** — describe your site and get a full theme with templates, template parts, patterns, and theme.json
- **AI design spec + deterministic template assembly** — the AI generates a design specification (colors, typography, copy, design narrative) and deterministic skeleton templates are assembled with those design tokens, ensuring reliable, valid output every time
- **8 curated template skeletons** — Starter, Editorial, Portfolio, Bold, Minimal, Magazine, Starter Plus, and Creative — each with distinct layout patterns and block structures
- **Live preview via WordPress Playground** — renders the generated theme in a real browser-based WordPress instance using WebAssembly, no server required
- **Zero `wp:html` blocks** — enforced by a stack-based block markup validator that catches nesting errors, forbidden blocks, and unknown block names
- **3 style variations per theme** — dark mode, high-contrast (WCAG AA), and an AI-designed variation named after the design narrative
- **Download as installable ZIP** — drop it into any WordPress 6.0+ site via Appearance > Themes > Upload
- **8 preset color palettes** — choose from Obsidian, Atelier, Nordic, Forest, Dusk, Ember, Linen, or Midnight, with full per-color customization
- **SSE streaming generation** — real-time progress updates as each phase completes (design spec, template assembly, validation, packaging)
- **AI-generated copy with voice** — hero headings, CTAs, feature descriptions, and 404 messages are generated by the AI to match the site's tone and personality, not generic placeholders
- **Prompt suggestion pills** — one-click preset descriptions for 7 site types (Photography, Boutique Store, Writer, Agency, Music, Documentation, Recipe Blog)
- **Hero style wireframe previews** — visual SVG diagrams showing Full Width, Split, Minimal, and None layout options before generating
- **120 tests passing** — unit tests covering validators, assembler, packager, AI layer, routes, and client routing

## Prerequisites

- Node.js 20+
- pnpm 10+
- A Gemini API key (default) or Anthropic API key

## Setup

```bash
git clone https://github.com/MandeeTopicz/wp-theme-generator.git
cd wp-theme-generator
pnpm install
cp .env.example packages/server/.env
# Add your API key to packages/server/.env
# Default provider is Gemini: set GEMINI_API_KEY
# Or use Anthropic: set AI_PROVIDER=anthropic and ANTHROPIC_API_KEY
```

## Running

```bash
# Terminal 1 — client
pnpm --filter @wp-theme-gen/client dev

# Terminal 2 — server
pnpm --filter @wp-theme-gen/server dev

# Open http://localhost:5173
```

## Architecture

### Monorepo Structure

```
packages/
  shared/    @wp-theme-gen/shared — types, validators, constants
  server/    @wp-theme-gen/server — Express API, AI layer, template skeletons, theme assembler
  client/    @wp-theme-gen/client — React + Vite frontend, Playground preview
```

**packages/shared** is the single source of truth for `ThemeManifest`, `BlockTemplate`, `ColorPaletteEntry`, `CopyStrings`, the `TEMPLATE_CATALOG`, and all validation logic. Both client and server import from `@wp-theme-gen/shared`, so type drift between frontend and backend is caught at compile time. The validators here enforce WordPress block markup rules: no `wp:html` blocks, proper comment nesting via a stack-based parser, block names checked against a `CORE_BLOCKS` allowlist, and theme.json structure validated with Zod schemas.

**packages/server** is an Express API with three active routes: `POST /api/generate` (AI design spec + deterministic assembly + ZIP), `POST /api/validate` (standalone validation), and `GET /api/download/:sessionId` (ZIP delivery + Playground fetch). The AI layer defines a provider interface (`AIProvider`) with a factory function that reads `AI_PROVIDER` from the environment, making provider swaps a config change. The template system contains 8 hand-crafted skeleton builders that produce valid WordPress block markup with color slot placeholders, which are resolved against the AI-generated color palette. The theme assembler builds `style.css`, `theme.json` (with styles block for proper background/text colors), template files (with skip-link + header/footer parts), pattern files (with PHP registration headers), and style variations.

**packages/client** is a React 18 + Vite app with Tailwind CSS. The multi-step form wizard collects theme requirements across 6 steps (description, identity, color palette, typography, layout, review). On submit, it navigates immediately to the result page (no waiting for the API) where a generation context tracks SSE progress. The result page has a two-panel layout: left sidebar (file tree, download, validation summary) and main content area with tabs (Live Preview via WordPress Playground, Design System, Validation). Files can be inspected in a slide-over panel with syntax highlighting.

### Generation Pipeline

The generation pipeline combines AI creativity with deterministic reliability:

1. **User input** flows through a 6-step form that collects site description, theme name/slug, color palette selection (8 presets with customization), typography choices, layout options (hero style, navigation, footer), and a template skeleton selection.

2. **Pass 1 (AI — Design Spec):** A single AI call to Gemini (or Claude) produces a `DesignSpec` — a JSON object containing: a color palette with semantic names, font families, layout dimensions, a design narrative describing the aesthetic in evocative language, and `CopyStrings` — all the human-readable text for the theme (hero heading, CTA copy, feature descriptions, 404 message, etc.). The design narrative is the key to non-generic output: it ensures the AI generates copy with a consistent voice rather than placeholder text.

3. **Pass 2 (Deterministic — Template Assembly):** The selected template skeleton (e.g., "editorial", "bold", "creative") is built with color slots resolved from the AI-generated palette. The `resolveColorSlugs` function analyzes the palette by luminance, saturation, and contrast ratio to assign semantic roles (background, text, surface, accent, muted). Then `interpolateCopy` replaces all `{{PLACEHOLDER}}` tokens in the skeleton with the AI-generated copy strings, producing complete template markup.

4. **Validation:** The assembled manifest is validated — slug format, theme.json schema, block markup nesting/allowlist via the stack-based parser, and WCAG contrast checks.

5. **Packaging:** The assembler builds `style.css` (with WordPress theme header and layout safety net CSS), `theme.json` v3 (with full color palette, typography, spacing, and styles block), wraps templates with skip-links and header/footer template-part references, wraps patterns with PHP registration headers, and generates 3 style variations. Everything is archived into a ZIP with correct WordPress theme directory structure.

6. **Delivery:** The ZIP and manifest JSON are stored in the OS temp directory keyed by session ID. The client receives the manifest and validation result via SSE, renders the result page, and optionally boots WordPress Playground to show a live preview.

### Why This Architecture

The original prototype used a two-pass approach where the AI generated both the design spec AND all block markup. This was unreliable — the AI would produce creative designs but with nesting errors, hallucinated blocks, or inconsistent color application. The current architecture splits the problem:

- **AI handles what it's good at:** creative decisions (color palettes, typography pairing, copy with personality, design narrative)
- **Deterministic code handles what it's good at:** valid block markup, proper nesting, consistent color application, correct WordPress template structure

This means every generated theme has structurally valid markup guaranteed by the skeleton templates, while the AI provides the creative differentiation through colors, typography, and copy. The trade-off is less layout variety (8 skeletons vs. infinite AI-generated layouts), but dramatically higher reliability.

## Known Limitations

- **Template variety is bounded by 8 skeletons** — unlike fully AI-generated markup, layout variety is constrained to the pre-built skeleton templates. Adding more skeletons is straightforward but requires manual block markup authoring.
- **Iteration/refinement is not yet supported** — the template-first architecture doesn't support "make the header darker" style modifications. Users regenerate with different options instead. This is the primary trade-off of deterministic assembly over AI-generated markup.
- **Style variations sometimes need regeneration** to get truly distinct palettes — the AI occasionally produces variations that are too similar to the base theme
- **WordPress Playground takes 15-30 seconds to boot** on first load — this is inherent to loading a full WordPress instance via WebAssembly in the browser
- **Generated templates use seeded sample content for preview** — real site content will look different; the preview is a demonstration of the theme's design, not the user's actual content
- **No WooCommerce template support yet** — `shop.html`, `single-product.html`, `cart.html`, and `checkout.html` are not generated
- **Rate limited to 5 generations per 15 minutes per IP** — this is a safety measure for API cost; production deployment would need a proper job queue
- **Font loading depends on system/Google Fonts availability** — generated themes reference font families by name but don't bundle font files; WordPress handles loading via theme.json font-face declarations
- **Color palette customization informs the AI** but the model may occasionally use slightly different shades than the exact hex values provided
